# Borrador BD - Iteracion 1

Este documento resume tablas candidatas para la base de datos del Hub a partir del draft general `index/login`.
No es DDL. No define tipos de datos, PK/FK ni restricciones tecnicas finales.
Su objetivo es servir como registro inicial para validar alcance funcional antes de modelar la base real.

TABLA: people
Finalidad:
Registrar personas de negocio o custodios vinculados a activos, entregas, recepciones y consultas.

Campos:
- id
  Observacion: Identificador interno de la persona dentro del Hub.
- full_name
  Observacion: Nombre visible de la persona para busqueda, actas y fichas.
- identifier
  Observacion: Identificador corporativo, legajo o codigo interno del colaborador.
- email
  Observacion: Correo principal usado en busquedas, contacto y cruce con iTop.
- area
  Observacion: Area o unidad organizacional de la persona.
- job_title
  Observacion: Cargo o funcion laboral mostrada en fichas e informes.
- manager_name
  Observacion: Nombre del jefe o responsable jerarquico, util para contexto operativo.
- location_id
  Observacion: Referencia a sede, oficina o ubicacion asociada a la persona.
- status
  Observacion: Estado de vigencia de la persona, por ejemplo activa o inactiva.
- hire_date
  Observacion: Fecha de ingreso, util para reportes de asignacion o dotacion.
- source_system
  Observacion: Sistema origen del dato, por ejemplo iTop o carga local.
- external_ref
  Observacion: Id externo para sincronizacion con sistemas maestros.
- created_at
  Observacion: Fecha de creacion del registro en Hub.
- updated_at
  Observacion: Ultima fecha de modificacion del registro.

Notas de iteracion:
- Este listado mezcla tablas muy probables con tablas que aun deben validarse.
- Algunas tablas pueden fusionarse en iteraciones futuras si decides simplificar.
- Algunas tablas pueden separarse mas si luego necesitas auditoria fuerte o alta trazabilidad.
- Para autenticacion, el draft sugiere acceso con credenciales de iTop; por eso no se propuso una tabla local de passwords como base principal.
- Antes de pasar a DDL conviene validar: origen maestro de personas, relacion real con iTop, numeracion documental, archivos, permisos, y si los reportes seran configurables o fijos.
